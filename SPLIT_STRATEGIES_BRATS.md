# Split strategies for BraTS-style datasets (`train.txt` / `val.txt` / `test.txt`)

Reference notes for choosing how to split subjects into train, validation, and test sets when a repo (e.g. M3AE) expects split text files.

---

## Practical options for split files

### 1. Use official challenge split / protocol (if available)

- **Why:** Most reproducible for paper-style comparison.
- **Caveat:** For BraTS **training** data, official **hidden** validation/test labels are not always available locally, so this option can be **limited** unless you use the challenge’s prescribed procedure (e.g. online evaluation for some benchmarks).

---

### 2. Reuse a published split from another paper or repo

- **Why:** Best for **apples-to-apples** comparison with that work.
- **Examples:** RFNet / M3AE split files if the authors publish `train.txt`, `val.txt`, `test.txt` (or equivalent) for the same dataset year and layout.

---

### 3. Patient-ID deterministic split (no RNG)

- **How:** Sort subject IDs and split by index (e.g. first **70%** train, next **10%** val, last **20%** test).
- **Why:** Fully **reproducible** without random seeds; easy to describe in a report.

---

### 4. Stratified split (if metadata is available)

- **How:** Split while preserving proportions of groups (e.g. **HGG vs LGG**, or a proxy for **site/scanner** if encoded in metadata).
- **Why:** Keeps **class / domain balance** more similar across train, val, and test than a naive random split.

---

### 5. k-fold cross-validation

- **How:** Generate fold files (`fold1_train.txt`, `fold1_val.txt`, …) and rotate which fold is validation/test.
- **Why:** **Stronger** evaluation and less dependence on a single lucky/unlucky split.

---

### 6. Group-wise split

- **How:** Split by **institution**, **scanner**, or **site** (when such grouping exists in metadata).
- **Why:** Tests **domain generalization** (model is evaluated on “unseen centers”), not just random patient mixing.

---

## M3AE loader: how split files are used (implementation reference)

This section records **how `get_datasets_brats20_rf` in `dataset/brats.py` maps `train.txt` / `val.txt` / `test.txt` to PyTorch datasets**, so you can align experiments with the code and write accurate methods text.

### Upstream-style behavior (typical M3AE / RFNet convention)

Many BraTS repos that ship three list files use this pattern:

| File        | Role in loader |
|------------|----------------|
| `train.txt` | Subject IDs used for **training** |
| `val.txt`   | Subject IDs **also merged into training** (same bucket as `train.txt`) |
| `test.txt`  | Subject IDs used for **validation** during training (often named “test” in filenames but acts as the **val** set in the training loop) |

So **training sees `train ∪ val`**, and **`test.txt` drives the validation loader** (e.g. `pretrain.py` → `l_val_dataset`). There is **no separate held-out test split** in that setup unless another script loads different lists.

### Local implementation in this workspace (true 70 / 10 / 20)

To match a standard **train / validation / held-out test** split (e.g. **deterministic 70% / 10% / 20%** by **sorted** `BraTS20_Training_*` folder names), the loader was updated to:

| File        | Role in loader |
|------------|----------------|
| `train.txt` | **Training only** — `full_train_dataset` |
| `val.txt`   | **Validation only** — `l_val_dataset` (metrics / early stopping) |
| `test.txt`  | **Held-out test** — listed for reproducibility; **not** loaded by `get_datasets_brats20_rf` for train or val (add a separate eval path if you need test-set metrics in code) |

Split files live next to the case folders, e.g. under  
`dataset/archive/BraTS2020_TrainingData/MICCAI_BraTS2020_TrainingData/`  
(one subject folder name per line, e.g. `BraTS20_Training_001`).

**Reproducing the fixed split:** sort all `BraTS20_Training_*` basenames; assign the first **⌊0.70·N⌋** to `train.txt`, the next **⌊0.10·N⌋** to `val.txt`, and the remainder to `test.txt` (for **N = 369** this yields 258 / 36 / 75).

---

## Recommended workflow (fast + defensible)

1. **Start** with one **deterministic fixed split** (e.g. **70 / 10 / 20** by **sorted** patient ID).
2. **After** the pipeline runs end-to-end, add **3-fold CV** (or **5-fold** if time and GPU allow).
3. **Report both:**
   - **Fixed-split** numbers (easy for others to reproduce the same lists), and  
   - **CV mean ± std** (more robust summary of performance).
